# 間食購入申請 (Snack Request) — Phase 1: Requirements

> 일본 사무실(7명, 단일 사무실)을 위한 주간 간식 신청·일괄 발주 시스템
> 한국 본사 HOTSELLER ERP `간식 구매 요청`을 모델로 단순화

---

## 1. 배경 / 목표

- 일본 사무실 직원이 매주 자유롭게 간식을 신청
- 발주 담당자(admin)가 매주 월요일 Amazon에서 일괄 발주
- 신청·집계·이력 관리를 ERP 내에서 일원화

## 2. 사용자 / 역할

| 역할 | 권한 |
|---|---|
| 일반 직원 (7명) | 신청 등록, 본인 신청 취소(마감 전), 고정 구매 등록·삭제·토글, 전사/내 이력 조회 |
| 발주 담당자 (`role='admin'`) | 위 권한 + 이번 주 일괄 "주문완료" 처리, 모든 신청 강제 취소(반려) |

> 별도 승인 단계 없음. 신청 = 즉시 "주문 대기(pending)"

## 3. 기능 요구사항 (Functional Requirements)

### FR-1. 간식 신청
- 직원은 마감 시점까지 자유롭게 신청 가능
- 입력 항목: Amazon 상품 링크, 제품명, 단가(¥), 수량, 비고(선택)
- 합계는 자동 계산 (단가 × 수량)
- "申請して続ける" 버튼으로 연속 신청 지원 (모달 유지, 폼만 리셋)
- 신청 즉시 상태 = `pending`

### FR-2. 마감 후 신청 자동 이월
- 마감(매주 금요일 24:00 JST) 이후 신청은 **자동으로 다음 주(차주 월요일)로 분류**
- DB의 `week_start` 필드는 신청 시각 기준으로 다음 주 월요일이 자동 계산됨
- UX: 마감 후 신청 시 "次週分として申請されます" 안내 문구 노출

### FR-3. 신청 취소
- 본인은 자기 신청 취소 가능 (`status='pending'` && 마감 전)
- 발주 담당자는 모든 신청 강제 취소 가능 (사유 메모 권장)
- 취소 시 `status='cancelled'`, 리스트에서 회색 처리(또는 숨김 토글)

### FR-4. 고정 구매 (매주 자동 신청)
- 직원이 기간(시작일 ~ 종료일) 지정해 등록
- 매주 월요일 00:05 JST에 활성 고정 구매를 그 주 신청에 자동 추가
- 시작일·종료일은 자동으로 그 주 월요일로 정렬
- 활성/비활성 토글 (비활성 시 자동 추가 중단, 데이터는 보존)
- 중복 방지: `(user_id, fixed_id, week_start)` 유니크

### FR-5. 리스트 / 조회
- **이번 주 전사 신청** (기본 뷰): 신청자 / 구매방법(=Amazon 고정) / 제품명(링크) / 수량 / 금액 / 상태
- **내 신청 이력** 토글 (월별 그룹)
- 합계(¥) 및 건수 표시
- 마감 D-day 표시 (`D-2`, `D-1`, `当日`, `締切`)
- 정렬: 신청 시각 내림차순

### FR-6. 누적 통계 카드
- 카드 1: **今月の私の累計** (금액 · 건수)
- 카드 2: **今月の全社累計** + **一人あたり平均** (= 전사 합 / 활성 직원 수)

### FR-7. 발주 담당자 액션
- 이번 주 신청 일괄 "주문완료(`ordered`)" 처리 버튼
- 처리 시 `ordered_at = now()` 기록, `status='ordered'`
- 이미 `ordered`인 항목은 건너뜀

### FR-8. 안내·반려 가이드 (모달 상단 고정)
- **마감/주문**: 締切=毎週金曜24:00 / 発注=毎週月曜
- **반려 대상**:
  - 冷凍·生鮮食品
  - 賞味期限の短い大量食品
  - スナック·飲料以外(健康食品·おもちゃ等)
  - 高級嗜好品(高級チョコ·高級ジャーキー等)
- **Amazon 바로가기 버튼**: `https://www.amazon.co.jp/`

## 4. 비기능 요구사항 (Non-Functional)

| 항목 | 요구사항 |
|---|---|
| 언어 | 일본어 UI 기본 + i18n 키 한국어도 동시 등록 (기존 ja/ko 토글 호환) |
| 인증 | 기존 ERP `ProtectedRoute` 재사용 |
| 권한 | 발주 담당자 = `users.role = 'admin'` |
| 시간대 | **Asia/Tokyo (JST)** — `week_start`, 마감 판정 모두 JST 기준 |
| DB | 기존 PostgreSQL에 마이그레이션 추가 |
| Cron | **node-cron 서버 내장** — 서버 부팅 시 자동 등록, 매주 월요일 00:05 JST |
| 성능 | 7명 규모, 부하 미미. 인덱스는 `(week_start)`, `(user_id)` 두 개로 충분 |

## 5. 범위 외 (Out of Scope)

- Amazon 자동 발주 연동 (담당자가 수동 주문)
- 결제·정산 (회사 카드 일괄 결제 가정)
- 도착 확인·검수 워크플로
- 그래프/차트 (한국판 층별·부서별 차트 미구현)
- Slack/메일 알림 (1차 출시 후 검토)
- 신청 한도 (1인당 월 ¥상한 등) — 추후 필요 시 추가

| 항목 | 결정 |
|---|---|
| 그래프·탭 | 모두 제거 (단일 리스트) |
| 마감/주문 사이클 | 한국과 동일 (금요일 24:00 마감 → 월요일 일괄 발주) |
| 고정 구매 | 포함 |
| 승인 흐름 | 없음 (신청 = 즉시 주문 대기) |
| 발주 담당자 권한 | `role='admin'` 재사용 |
| 마감 후 신청 | 다음 주로 자동 이월 |
| Cron | node-cron 서버 내장 |

## 8. 완료 기준 (Acceptance Criteria)

- [ ] 직원이 `/erp/snack-request`에서 신청 모달로 신청 → 즉시 리스트에 반영
- [ ] 마감 D-day 표시가 JST 기준으로 정확히 동작
- [ ] 마감 후 신청이 다음 주로 자동 분류
- [ ] 고정 구매 등록 후 다음 월요일 00:05에 자동으로 그 주 신청에 추가됨
- [ ] 본인 신청은 마감 전까지 취소 가능
- [ ] admin이 일괄 "주문완료" 클릭 시 이번 주 pending 항목 모두 ordered 처리
- [ ] 통계 카드(이번 달 내 누적 / 전사 누적·평균)가 정확히 계산됨
- [ ] 일본어 UI 전체 번역 완료, 한국어 토글 시 한국어로 표시됨
